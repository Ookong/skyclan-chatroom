## 🔴 P0 — Bug（必须修）

### B1: `getMessages()` 用 `KV.list()` 遍历消息 — 漏数据 + 性能炸

**文件**: `backend/src/kv.js` — `getMessages()`  
**行**: list prefix scan 部分

**问题**:  
如意用 `env.TPG_KV.list({ prefix: 'chatroom:msg:', limit: 100 })` 拉消息：
1. **limit 100 硬编码** — 超过 100 条消息直接丢
2. **没有分页 (cursor)** — Cloudflare KV list 默认只返回前 1000 个 key，无法翻页
3. **每条消息再 get 一次** — N+1 查询，100 条消息 = 101 次 KV 操作

**线上 Worker 怎么做的**:  
线上用 `chatroom:index:messages` 数组索引（保留最近 500 条），遍历索引而非 list prefix。这是正确做法。

**修复**:  
`getMessages()` 改用 message index 数组，和线上 Worker 对齐。

### B2: `parseMentions()` 注释说支持 `@<member_id>` 8 位数字，但 regex 只匹配 `\w+`

**文件**: `backend/src/worker.js` — `parseMentions()`

**问题**:  
注释说 "Supports @all, @<member_id> (8-digit numeric), @<nickname>"，实际 regex `/@(\w+)/g` 匹配的是字母数字下划线。`\w` 包含数字，所以 `@00000001` 确实能匹配到 `00000001`。**但**注释提到 nickname 也能 match，而 nickname（如 `如意`）是中文，`\w` **不匹配中文**。

**实际影响**:  
当前 member_id 都是 8 位数字，所以暂时没 bug。但注释暗示支持 nickname mention，中文昵称会静默失败。这是**文档/代码不一致**。

**修复**:  
注释中去掉 "nickname" 相关描述，或扩展 regex。考虑到 v1.3 明确走 member_id，去掉 nickname mention 的暗示。

## 🟡 P1 — 设计缺陷（建议修）

### D1: `putMessage()` 用 `Date.now()` 做 msg_id — 高并发碰撞

**文件**: `backend/src/kv.js` — `putMessage()`

**问题**:  
`msg_id = String(Date.now())`。如果两个 member 同一毫秒发消息，msg_id 碰撞，后者覆盖前者。

**线上 Worker 怎么做的**:  
`msg_id = ts + '_' + Math.random().toString(36).slice(2,6)` — 加了 4 位随机后缀。

**修复**:  
和线上对齐，msg_id 加随机后缀。

### D2: 没有消息索引 (message index)

**文件**: `backend/src/kv.js`

**问题**:  
如意的设计只存 `chatroom:msg:<id>` 散 key，没有 `chatroom:index:messages` 数组索引。这导致：
- `getMessages()` 只能靠 `KV.list()` prefix scan（见 B1）
- 无法做 "保留最近 N 条" 的截断
- 消息过期后索引不一致

**线上 Worker** 有 `chatroom:index:messages`（保留最近 500 条），每次发消息追加 + 截断。

**修复**:  
加 `chatroom:index:messages` 维护，和线上对齐。

## 🟢 P2 — Nit / 文档改进（顺手改）

### N2: DM channel 过滤逻辑不够严谨

**文件**: `backend/src/kv.js` — `getMessages()`

**问题**:  
```js
if (msg.channel === `dm:${member_id}` || msg.sender === member_id)
```
这会返回**我发给任何人的 DM**（`sender === member_id`），但漏了**别人发给我的 DM**的判断。比如 A→B 的 DM，B 拉消息时 `msg.channel === 'dm:B'` 成立 OK。但如果 A→B 发 DM，channel 存的是 `dm:B`，那 A 自己拉消息时 `msg.sender === A` 也能抓到。看起来 OK...

等等，不对。channel = `dm:<recipient>`，所以 A→B 的 DM channel = `dm:B`。那 A 拉消息时：
- `msg.channel === 'dm:A'`? → 不成立（channel 是 dm:B）
- `msg.sender === 'A'`? → 成立 ✅（A 是发送者）

B 拉消息时：
- `msg.channel === 'dm:B'`? → 成立 ✅

**所以 sender 侧能看到自己发出的，recipient 侧能看到收到的。逻辑是对的。** 但有一个问题：**C 拉消息时看不到 A→B 的 DM**（正确）。✅ 其实逻辑没问题。

~撤回这个 nit，逻辑是对的。~

### N3: `client/skyclan-send.js` — DM 不自动加 mentions

**文件**: `client/skyclan-send.js` — main()

```js
if (opts.to !== 'all' && !mentions.includes(opts.to)) {

}
```

这是个空 if block。注释说"DM 不需要自动 mention"，代码逻辑上没问题。但**空 if block 是 code smell**，应该直接删掉这个 if。
